# ScreenSeekeR Visual Grounding System

A production-quality implementation of the **ScreenSeekeR** recursive visual
search architecture (from the ScreenSpot-Pro paper) for locating arbitrary
desktop icons and GUI elements — demonstrated by locating and automating a
**Notepad** desktop shortcut regardless of where it sits on screen.

This is **not** a `screenshot -> ground -> click` script. It is a planner-guided,
recursive, self-verifying search pipeline.

## Architecture

```
                ┌───────────────────────────────────────────────────────┐
                │                     ScreenSeekeR                      │
                │                                                       │
 Screenshot ───►│  PLANNER (Gemini 2.5 Flash)                           │
   (MSS)        │   position inference → ranked candidate regions       │
                │   + neighbor anchors (e.g. "near Recycle Bin")        │
                │            │                                          │
                │            ▼                                          │
                │  RECURSIVE SEARCH (priority order, depth ≤ MAX_DEPTH) │
                │   crop region (OpenCV)                                │
                │            │                                          │
                │            ▼                                          │
                │  GROUNDER (Gemini Robotics ER 1.6 Preview)            │
                │   point prediction on the SMALL crop only             │
                │            │                                          │
                │            ▼                                          │
                │  VERIFIER (Gemini 2.5 Flash)                          │
                │   result checking on annotated context patch         │
                │     ├─ reject → next region / recurse deeper          │
                │     └─ accept ▼                                       │
                │  COORDINATE PROJECTION                                │
                │   crop-local → parent → … → global screen pixels      │
                └────────────┬──────────────────────────────────────────┘
                             ▼
                  EXECUTOR (PyAutoGUI) — smooth move + double-click
```

## Why ScreenSeekeR (and not direct grounding)?

Grounding models lose precision on large, dense screenshots: a desktop icon is
a tiny fraction of a 4K frame. ScreenSeekeR's insight is that a strong
*planner* VLM can act as a search guide for a precise but myopic *grounder*:

1. **Position inference** — the planner uses GUI layout knowledge to predict
   *where* the target probably is (desktop icon grids, taskbar, neighbors),
   even when the target itself is barely visible.
2. **Candidate regions** — multiple ranked regions with neighbor anchors.
3. **Recursive narrowing** — failed regions are re-planned at higher zoom;
   each recursion level grounds against a smaller, sharper crop.
4. **Result checking** — every grounding point is independently verified on an
   annotated context patch before any click happens; rejects trigger
   backtracking to the next candidate.

This implementation maps those ideas 1:1 onto `src/planner.py`,
`src/screenseeker.py`, `src/grounder.py`, and `src/verifier.py`. The prompts in
`prompts/` are adapted from the paper's Position Inference and Result Checking
prompts.

No template matching, stored icon images, OCR pipelines, or hardcoded
coordinates are used anywhere. Every iteration of the automation task performs
a **fresh screenshot and fresh grounding**.

## Setup

Requires Python 3.11+ and [uv](https://docs.astral.sh/uv/).

```bash
uv sync          # creates .venv and generates uv.lock
export GEMINI_API_KEY=your_key_here    # PowerShell: $env:GEMINI_API_KEY="..."
```

> `uv.lock` is machine-generated by `uv sync` / `uv lock` (it contains resolved
> hashes and cannot be authored by hand). Commit it after the first sync.

## Run

```bash
uv run python main.py --locate-only          # locate Notepad, save artifacts, no clicks
uv run python main.py                        # full task: 10 posts via Notepad
uv run python main.py --posts 3
uv run python main.py --instruction "the Recycle Bin icon on the desktop" --locate-only
uv run pytest                                # unit tests (no API calls)
```

The automation task: locate Notepad → double-click → verify it launched →
download 10 posts from `jsonplaceholder.typicode.com` → write each as
`Title: {title}\n\n{body}` → save to `~/Desktop/tjm-project/post_{id}.txt` →
close Notepad → **re-ground from a fresh screenshot** before the next post.

## Debug artifacts

Every run produces a full visual audit trail:

```
debug/
├── full_screenshot.png                      # raw capture
├── plan_depth0.json                         # planner regions + priorities + neighbors
├── crop_depth0_attempt1.png                 # candidate region crop
├── crop_depth0_attempt1_annotated.png       # green box + crosshair on grounded point
├── verification_depth0_attempt1.png         # patch shown to the verifier
├── crop_depth1_attempt1.png                 # recursion artifacts
├── search_trace.json                        # full attempt log (depth/region/verdicts)
├── final_result.png                         # full screen + region + click point
└── run.log
```

Annotated detections are also written to `output/{zone}_detection.png`
(`top_left_detection.png`, `center_detection.png`, `bottom_right_detection.png`,
…) based on where the icon was actually found — move the icon and re-run to
collect all three required screenshots.

## Robustness

- **Retries**: `MAX_RETRIES = 3`, 1 s delay — applied to planner, grounder,
  verifier, the posts download, and whole-search "icon not found" failures.
- **Malformed JSON**: tolerant extraction (fenced blocks, prose-embedded JSON,
  robotics `[{"point":[y,x]}]` format) with safe fallbacks.
- **Multiple matching icons**: neighbor anchors in planning + verifier
  rejection of look-alikes drive the search to the correct instance.
- **Partially obscured icons / varied wallpapers**: pure semantic VLM search,
  no pixel templates to break.
- **Offline / timeouts**: every network call is bounded and retried; the run
  fails with a clear error instead of clicking blindly.
- **Safety**: PyAutoGUI failsafe enabled (cursor to corner aborts).

## Known failure cases

- The verifier can produce false rejections on very small icons, costing extra
  recursion (mitigated by the parent-image context patch).
- Windows 11 Notepad's tabbed UI can vary across builds; Save As flow uses
  `Ctrl+Shift+S` and pre-deletes target files to avoid overwrite dialogs.
- DPI scaling: MSS captures physical pixels while PyAutoGUI may use logical
  pixels on scaled displays; run at 100% scaling or mark the process DPI-aware.
- Identical duplicate shortcuts with identical labels are inherently ambiguous.

## Limitations & future improvements

- Single-monitor (primary) capture; multi-monitor projection is a natural
  extension of the origin-chain utility.
- Planner re-planning at depth currently reuses the same prompt; a dedicated
  "zoomed-in re-plan" prompt with parent-region memory would match the paper
  even more closely.
- Could add an ensemble of grounding samples with spatial clustering, and a
  cache of *region priors* (not coordinates) to warm-start planning.
- An accessibility-tree fallback (UIA) would complement pure vision.
